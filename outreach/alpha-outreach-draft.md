## A. Customer angle (AI SDR / assistant / support-agent founders)

**Subject:** When a token expires mid-run

Hi {{First Name}},

Quick question about {{Company}}. When one of your agents is partway through a
run and the user's Google or Microsoft token expires, what happens today? In
most setups the run fails quietly, and reconnecting doesn't repair the run that
was already in flight. You end up restarting from scratch, and if something like
a send or a write had already partly happened, there's a real risk of it running
twice.

I've been building Revive to solve exactly that. It detects the token failure,
saves the exact step the run was on, sends the user a one-time reauthorization
link, and resumes the same run once access is restored. It also reconciles side
effects so nothing fires twice.

I'm looking for a handful of alpha users to test it properly. I'm not selling
anything. I'll handle the setup myself, and all I'm hoping for is honest
feedback. Would you have 20 minutes this week?

Thanks,
Samy

## B. Partner angle (Nango / Arcade / Composio)

**Subject:** The step after "connection needs reauth"

Hi {{First Name}},

{{Company}} handles token lifecycle really well, right up to the point where a
refresh token dies. After that, the practical answer is usually to flag the
connection, pause syncs, and ask the user to reconnect.

Revive picks up from there. It takes that dead-connection signal, finds the run
that depended on it, and splices the new token back in so the run resumes
instead of starting over. We already work alongside your reconnect session flow.

I think of it as a complement rather than a competitor. Your customers stay with
you and get run-level resume on top. Would you be open to a short call to see
whether there is a partnership worth exploring?

Thanks,
Samy
